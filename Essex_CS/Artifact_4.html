What is ReDOS and what part do ‘Evil Regex’ play?

ReDOS (Regular expression Denial of Service) is a Denial of Service attack. According to Weidman, A.(n.d), the attack exploits the extreme situations that most Regular Expression implementations may reach which cause them to work very slowly. An attacker causes a program using a Regex to enter these extreme situations making the program hang for a long time.

The Regex pattern that can get stuck on crafted input is called Evil Regex. Evil Regex contains grouping with repetition and repetition / alternation with overlapping inside the repeated group. (eg. (a+)+, ([a-zA-Z]+)* & (.*a){x} for x\>10 ) An attacker looks for applications that use Regex containing an Evil Regex and sends a well-crafted input. Alternatively, the attacker can inject an Evil Regex if a Regex itself is affected by a user input.

What are the common problems associated with the use of regex? How can these be mitigated?

There are four common pitfalls of the Regex (Regex.info, n.d.):

  • Catastrophic Backtracking: It is likely to be a case of Catastrophic Backtracking when your regular expression seems to take forever or crashes your application. The possible solution is to be more specific about what you want to match
  • Making Everything Optional: Regex will match a zero-length string anywhere if all the parts of Regex are optional. Solution is that you need to express the facts that different parts are optional depending on which parts are present
  • Repeating a Capturing Group vs Capturing a Repeated Group: it is a common mistake when creating a regular expression which needs a capturing group to grab part of the text matched. The capturing group is to be repeated instead of the repeated group. The repeated capturing group will capture only the last iteration, while a group capturing another group that’s repeated will capture all iterations.
  • Mixing Unicode and 8-bit Character Codes: using 8-bit character codes like \x80 with a Unicode engine and the subject string may give unexpected results
    • There are some regex best practices suggested by Li (2020):
  • Input validation: uses whitelist instead of black list when filtering such as file types and IP addresses.
  • Regex patterns should not be published: publish the patterns makes it easier for attackers to find security holes in regex pattern
  • Use validated patterns: use validated and secured regex pattern online is better choice than writing your own regex patterns for common use-cases
  • Multiple defense layers: In addition to rigorous input validation, other methods such as prepared-statement, least privileges principle and hashing can also be implemented
  • Testing: rigorously testing the application with unexpected and illegal input

    • How and why could regex be used as part of a security solution?

    “RegEx” is a special string that describes a search pattern, which is powerful in pattern matching. RegEx can be thought of having two parts including constants and operators. Constants are sets of strings while operators are symbols which denote the operations over these constants. Logile Analysis is an example of Regex being useful to make searching more quickly and accurate.

    As Li, V. (2020) mentioned, there are several security usages of RegEx:

  • Firewall Rules and Input Validation: RegEx can be used to specify firewall rules to block requests to certain file types by matching any request that contains the terms such as “exe”, “json”, “tar” and “rar”.
  • Proxy Rules: RegEx can be useful in application debugging and testing. RegEx can be used to filter traffic on debugging proxies which is more efficient
  • Malware Scanning: RegEx patterns is used in YARA to detect specific text or binary patterns in files that might indicate that the file is malicious
