Secure Software Development (Computer Science)


Module 3

Unit 1 - Introduction to Secure Software Development

This week, the lecturecast has brought me through the brief information and basics in the problems in cybercrime, weakness and proactive controls advised by the OWASP, integration of security measures in Agile model and the usage of UML diagrams. In the discussion forum, I read some in-depth investigations on the most possible software weaknesses identified by OWASP written from my classmates.

With little experience in programming after participating in a boot camp since February, I started working in an IOT company this month. During this short period of time doing practical programming, the developing environments I am in are totally functional driven with less security concerns involved. In boot camp, it is all about how I can make some fancy-looking stuff within a short period of time or a agile development model without security concerns. In my new company, most of the IOT data collecting and processing are not private or personal data and thus less priority in security. This unit has broadened my horizon especially in terms of the software security and the possible weaknesses.

Unit 2 - UML Modelling to Support Secure System Planning

I continue reading the book “Software Architecture with Python” in chapters 1 and 6. In the Chapter 1 Principles of Software Architecture, I got a glimpse of the importance of early stage design of software architecture, knowing that the architectural quality can be observed by various attributes like modifiability, testability, scalability, performance and the one which is also the protagonist of this module, security. In Chapter 6 Writing Secure Code, I got familiar with some security concerns in writing Python such as:

  • Reading input: use raw input over the input in order to return input as plain text instead of evaluating Python expression. Besides, getpass is a good library to perform validations on the returned data
  • Evaluating expression: avoid using eval and exec
  • Serialization: avoid using pickle / cPickle
  • Overflow error: Python does not suffer from pure buffer overflow errors, but for overridden __len__ methods on classes, guard should be implemented against integer overflows by using exception or TypeError handler
  • String formatting: prefer newer and safer format of template strings over older %s

Also in this Secure Code chapter, I found it interesting in studying race conditions. Python provides an excellent set of threading primitives. Race conditions and deadlocks are avoided by the threading primitives set to synchronize access to resources, such as threading.Lock. I had less concept in the topics related to condition, since at the moment most of my practical programming experience was in boot camp in which I was using Node.js in Typescript. But then I also found that although Node.js is a single threaded application, it is still possible to run into race conditions. The reason is that the program workflow includes I/O (database, file system) running on many threads. It is possible that the shared resource is outside of the Node.js being accessed or operated at the same time.

Brief reasons for the failure

  • The information systems were not adequately segmented from one another, thus the attackers were able to move from the web portal to other servers.

Impact of the failure

  • 143 million people was potentially affected
  • Personal information (names, addresses, dates of birth, Social Security numbers, drivers' licenses numbers) of more than 40% of the population of the United States was exposed
  • The upper ranks of Equifax's C-suite rapidly turned over

Human factor of the failure
Not only the system or server structural vulnerability make it convenient for the attacker, factors of management or mechanism actually involved a lot in the information system failure. The unpatched server vulnerabilities since the administrator missed the email from the CISO, the lack of formal process in place for vulnerability patching and the lack of authorities with mechanisms to auto-renew the digital certificates.
