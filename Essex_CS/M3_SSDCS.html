Unit 1 - Introduction to Secure Software Development

This week, the lecturecast has brought me through the brief information and basics in the problems in cybercrime, weakness and proactive controls advised by the OWASP, integration of security measures in Agile model and the usage of UML diagrams. In the discussion forum, I read some in-depth investigations on the most possible software weaknesses identified by OWASP written from my classmates.

With little experience in programming after participating in a boot camp since February, I started working in an IOT company this month. During this short period of time doing practical programming, the developing environments I am in are totally functional driven with less security concerns involved. In boot camp, it is all about how I can make some fancy-looking stuff within a short period of time or a agile development model without security concerns. In my new company, most of the IOT data collecting and processing are not private or personal data and thus less priority in security. This unit has broadened my horizon especially in terms of the software security and the possible weaknesses.

Unit 1 - Artifacts:

  • Collaborative Discussion 1: UML flowchart

    • Unit 2 - UML Modelling to Support Secure System Planning

    I continue reading the book “Software Architecture with Python” in chapters 1 and 6. In the Chapter 1 Principles of Software Architecture, I got a glimpse of the importance of early stage design of software architecture, knowing that the architectural quality can be observed by various attributes like modifiability, testability, scalability, performance and the one which is also the protagonist of this module, security. In Chapter 6 Writing Secure Code, I got familiar with some security concerns in writing Python such as:

    • Reading input: use raw input over the input in order to return input as plain text instead of evaluating Python expression. Besides, getpass is a good library to perform validations on the returned data
    • Evaluating expression: avoid using eval and exec
    • Serialization: avoid using pickle / cPickle
    • Overflow error: Python does not suffer from pure buffer overflow errors, but for overridden __len__ methods on classes, guard should be implemented against integer overflows by using exception or TypeError handler
    • String formatting: prefer newer and safer format of template strings over older %s

    Also in this Secure Code chapter, I found it interesting in studying race conditions. Python provides an excellent set of threading primitives. Race conditions and deadlocks are avoided by the threading primitives set to synchronize access to resources, such as threading.Lock. I had less concept in the topics related to condition, since at the moment most of my practical programming experience was in boot camp in which I was using Node.js in Typescript. But then I also found that although Node.js is a single threaded application, it is still possible to run into race conditions. The reason is that the program workflow includes I/O (database, file system) running on many threads. It is possible that the shared resource is outside of the Node.js being accessed or operated at the same time.

    • Unit 4 - Exploring Programming Concepts

    I continue reading the book “Software Architecture with Python”. I enjoyed reading the Chapter 7 Design Patterns in Python. I felt the power of structural design pattern with the intricacies of combining classes or objects into larger meaningful structures, providing the architect and developer with reusable behaviors. I have experimented with some patterns in Visual Studio Code. I went through the basic Singleton pattern first, which ensures just a single actual instance at a specific memory location which is accessible via a well-known access point. Along the process, I drilled into understanding the reasons of some usage of the dunder methods (eg. using __new__ to intercept and modify the arguments before the immutable object is created & using __call__ in Singleton metaclass which invoked when an instance of a class is used as a callable. ) and the reason of the meta-class version of Singleton inheriting from “type”, knowing that type is in fact a built-in metaclass used by Python to create all classes behind the scenes. I also tried other patterns, including builder pattern, factory pattern and prototype pattern as well. I will investigate more usage of different patterns in the future.

    • Unit 5 - An Introduction to Testing

    This week I was mainly focusing on studying the testing of software. I read Chapter 3 of the book “Software Architecture with Python”. Since the start of my programming journey from February, I have not done any test cases for all the projects I participated in bootcamp and the current project at work. Thus, I did not have clear concepts on the implementation of test cases, its advantages and even think it may not be necessary and time consuming to write.

    By reading this chapter, I start understanding the importance of testing, especially in terms of maintainability and security. Testability of a software system is a good indicator of quality, complexity, stability and predictability of a software system. I found that it is also a good mindset to think about how to make the code easily being tested while implementing the functionalities. In pursuing to write comprehensive unit tests also reduces the final coupling level of the codes.

    I also do some reading and discussion with teammates about the Cyclomatic Complexity. More details were recorded in the following.

    Unit 10 - From Distributed Computing to Microarchitectures

    From the lecturecast this week, I realized the advantages of a distributed system in microarchitectures. Compared to the monolithic architecture, distributed systems are a collection of individual components on different machines rather than being a single software product. Possible advantages includes: can utilise small specialist teams, working well with agile development approach, possible in using different languages or framework for different services, higher maintainability and significantly less prone to whole application system down. Some extra reflection was conducted about the Service Oriented Architecture Ontology. More details were recorded in the following.
